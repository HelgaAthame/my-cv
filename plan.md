### Как безопасно дать доступ — токены, а не пароли

Автор пыталась дать доступы, но я их убрала из файла (см. предупреждение в
начале раздела) и не могу использовать в том виде, в котором они были даны:

- **"Подключаюсь через GitHub" (Vercel, Supabase, Neon)** — это OAuth-логин
  через браузер: он требует, чтобы САМА автор была залогинена в GitHub в
  браузере и подтвердила вход (иногда с 2FA). У меня нет её пароля от
  GitHub и не должно быть — я работаю headless, без её живой браузерной
  сессии, поэтому "войти через GitHub" за неё технически невозможно
- **MongoDB Atlas по паролю** — дан был только пароль без логина/email, но
  даже с обоими вводить учётные данные аккаунта через браузер рискованно и
  не нужно — для программного доступа у всех этих сервисов есть API-токены,
  тот же паттерн, что и с `gh` (Personal Access Token вместо логина)

**Что реально нужно** — сделать один раз в браузере (по одному клику через
GitHub OAuth, это нормально и безопасно, т.к. это её собственное действие в
её сессии), затем сгенерировать токен и прислать только его:

- [x] **Vercel** — токен получен
- [x] **MongoDB Atlas** — API key (public + private) получен
- [x] **Supabase** — токен получен
- [x] **Neon** — проект и API key получены (уже с готовой Postgres-базой,
      её можно использовать вместо создания новой на News App backend)

Токены получены в чате, не записывались в файлы репозитория, использованы
для передеплоя Music Library backend и News App backend (см. выше — оба
починены).

Если вставляете токен прямо в чат — это нормально, я использую его сразу и
не буду записывать в файлы репозитория. Просто не надо класть секреты в
`plan.md` или другие файлы, которые лежат в git.
